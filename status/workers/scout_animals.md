## 2. Animal Lifecycle

### Buying
1. Player opens the **AnimalShop** (Marnie's).
2. Economy domain validates gold, deducts it, emits `ShopTransactionEvent { item_id, is_purchase: true }`.
3. `handle_animal_purchase` receives the event; calls `item_to_animal()` to resolve the kind.
4. Building check: coop animals → `animal_state.has_coop`; barn animals → `animal_state.has_barn`; companions → always allowed.
5. Entity is spawned with `Animal { age: Baby, days_old: 0, happiness: 128, … }` + `WanderAi` + sprite + name-tag child.
6. A `PlaySfxEvent` fires the `{item_id}_cry` sound.

### Spawning (technical)
- Spawned directly in `handle_animal_purchase` via `commands.spawn(…)`.
- Spawn position is a random point inside the pen rectangle.
- No separate "spawn" event path; spawning is synchronous inside the purchase handler.

### Feeding
- Player must have **hay** in hotbar and interact near the feed trough.
- The inventory/player domain consumes the hay and fires `ItemRemovedEvent { item_id: "hay" }`.
- `handle_feed_trough_interact` sets `fed_today = true` on **all** productive barn/coop animals simultaneously (not one-at-a-time).
- Companions (Cat, Dog, Horse) are explicitly skipped.
- Multiple hay events in one day are idempotent (setting `true` twice is harmless).

### Petting
- Player presses the tool-use key within 32 px of an animal **that has no product ready**.
- `handle_animal_interact` sets `petted_today = true` and `happiness += 5` (saturating).
- If already petted today, shows "Already happy!" text with no stat change.
- Petting is processed at most once per key-press (breaks on first in-range animal).

### Product Production
- Only **Adults** produce (`age == AnimalAge::Adult`).
- Products are set up at day-end by `handle_day_end_for_animals` (see §3).
- Collection: player presses tool-use key within 32 px of an animal with `product_ready == true`; `handle_product_collection` fires `ItemPickupEvent` and clears the flag.
- Quality is stored in `PendingProductQuality` component from day-end; read on collection.

| Animal | Product item_id | Frequency |
|---|---|---|
| Chicken | `egg` | Every day |
| Cow | `milk` | Every day |
| Sheep | `wool` | Every 3 days |
| Goat | `goat_milk` | Every 2 days |
| Duck | `duck_egg` | Every 2 days |
| Rabbit | `rabbit_foot` | Every 4 days |
| Pig | `truffle` | Every day (if happiness ≥ 50) |
| Horse, Cat, Dog | — | Never (companions) |

## 3. Day-End Processing

`handle_day_end_for_animals` fires on each `DayEndEvent` and processes every `Animal` entity:

### Step 1 — Consecutive Unfed Days (UnfedDays component)
- If `fed_today`: reset `UnfedDays.count = 0`.
- Else: `count = prev_count.saturating_add(1)`.
- Component is inserted (not updated) if missing, inserted each day otherwise.

### Step 2 — Happiness Adjustments
- Fed today: `happiness += 5` (saturating, max 255).
- Not fed: `happiness -= 12` (saturating, floor 0).
- Petted today: `happiness += 5` on top.
- Toast warnings:
  - `unfed_count == 3`: "hasn't eaten in 3 days and stopped producing!"
  - `unfed_count == 1 && happiness < 50`: "is hungry and unhappy…"

### Step 3 — Flag Reset
- `fed_today = false`, `petted_today = false`.

### Step 4 — Aging
- `days_old += 1` (saturating).
- If `age == Baby && days_old >= 5`: promote to `Adult`, fire toast + info log.

### Step 5 — Product Generation
Conditions for generating a product:
- `age == Adult`
- `fed_today_this_cycle` (new_unfed_count == 0)
- Not production-blocked (`!(not_fed_today && prev_unfed_count >= 3)`)
- `happiness > 0`

Quality thresholds (deterministic, no RNG):
| happiness | Quality |
|---|---|
| ≥ 230 | Iridium |
| ≥ 200 | Gold |
| ≥ 128 | Silver |
| < 128 | Normal |

Quality uses **post-adjustment** happiness (feeding/petting bonuses already applied before quality is computed).

### Step 6 — Resource Sync
`sync_animal_state_resource` in rendering.rs re-syncs `AnimalState.animals` every frame; day-end does not manually sync.

## 5. Can the Player Actually Buy Animals?

### Animal Shop listings (`src/data/shops.rs:153`)
The `AnimalShop` contains only **three** animal listings:

| item_id | Price |
|---|---|
| `animal_chicken` | 800g |
| `animal_cow` | 1,500g |
| `animal_sheep` | 2,000g |

**Missing from the shop**: `goat`, `duck`, `rabbit`, `pig`, `horse`, `cat`, `dog` — **7 of 10 kinds are not for sale.**

### Item-ID mismatch (critical bug)
The shop uses `item_id = "animal_chicken"` (prefixed with `animal_`), but `item_to_animal()` in `spawning.rs` matches on `"chicken"` (no prefix). The comment in `shops.rs` says `items prefixed "animal_" trigger animal spawning` (processed by the economy domain), but `handle_animal_purchase` receives a raw `ShopTransactionEvent` and calls `item_to_animal(&ev.item_id)` directly. **If the economy domain strips the `animal_` prefix before firing the event, this works; if it passes the full `"animal_chicken"` string, `item_to_animal` returns `None` and no animal is ever spawned.** This handoff is not verified within the animals domain and represents a potential silent failure.

### Cat and Dog
`item_to_animal` maps `"cat"` and `"dog"` but neither appears in any shop listing. There is no other purchase path visible in the animals domain.

## 6. Bugs and Gaps

### BUG-01 — `WanderAi.speed` match is non-exhaustive for 5 kinds (`spawning.rs:282`)
```rust
speed: match kind {
    AnimalKind::Chicken => 20.0,
    AnimalKind::Cow    => 14.0,
    AnimalKind::Sheep  => 16.0,
    AnimalKind::Cat    => 24.0,
    AnimalKind::Dog    => 28.0,
    // Goat, Duck, Rabbit, Pig, Horse — NO ARMS
}
```
Rust requires exhaustive matches. This compiles today **only because those five kinds can never reach this code** (they have no `item_to_animal` mapping and no shop entry). The moment a goat/duck/etc. is added to the shop this will be a **compile error**. A wildcard default speed should be added.

### BUG-02 — `item_to_animal` is incomplete (`spawning.rs:92`)
Goat, Duck, Rabbit, Pig, Horse are not mapped. Any `ShopTransactionEvent` for those kinds is silently dropped (`None` → `continue`). No warning is logged; the player loses gold with nothing to show for it.

### BUG-03 — Shop item-ID prefix mismatch (potential silent failure)
Shop entries use `"animal_chicken"` etc., but `item_to_animal` matches `"chicken"`. If the economy domain passes the raw `item_id` without stripping the `"animal_"` prefix, zero animals can ever be spawned. The animals domain provides no defensive logging for this case.

### BUG-04 — `SheepWoolCooldown` component reused for all multi-day producers
Goat, Duck, Rabbit, and Pig (for Pig it is unused) all share `SheepWoolCooldown.days_since_last_wool`. The `wool_cd` binding in the day-end query is `Option<&SheepWoolCooldown>` — meaning all those animals share a single cooldown component type. If a Goat and a Sheep both exist on the same entity (impossible in practice but a footgun), or if an animal is expected to have separate wool and milk cooldowns (e.g. a future kind that produces two products), this design breaks. The name is also misleading for non-sheep animals.

### BUG-05 — Production block logic inversion (`day_end.rs:183`)
```rust
let production_blocked = !fed_today_this_cycle && prev_unfed_count >= 3;
```
An animal blocked by 3+ unfed days will have `fed_today_this_cycle == true` the day it is finally fed (streak resets to 0). Since `!true == false`, `production_blocked = false`, so the block is lifted and the animal **produces on the very day it is fed again**. This is stated as intentional in the comment, but it means a 3-day starvation block evaporates immediately on the first feeding, which may not match the game design intent of a grace period.

### BUG-06 — `UnfedDays` component never initialized on spawn (`spawning.rs`)
Animals are spawned without the `UnfedDays` component. The day-end handler uses `Option<&mut UnfedDays>` and inserts it on first access. This is mostly fine, but `prev_unfed_count` will always be `0` the first day an animal goes unfed (missing component → defaults to 0), so the very first unfed day is silently invisible to the warning system.

### BUG-07 — Cat and Dog purchasable in code but absent from shop
`item_to_animal` returns `Some(AnimalKind::Cat)` for `"cat"` and `Some(AnimalKind::Dog)` for `"dog"`, but neither string appears in any shop listing. Players can never obtain cats or dogs through normal gameplay.

### BUG-08 — Horse missing from both shop and `item_to_animal`
Horse has full support (pen bounds, visual, name pool, interactions, day-end handling) but cannot be acquired through any visible purchase path.

### GAP-01 — No SFX assets verified for animal cries
The purchase handler fires `PlaySfxEvent { sfx_id: format!("{}_cry", ev.item_id) }`. For the shop item `"animal_chicken"` this becomes `"animal_chicken_cry"`, and for the `item_to_animal` path it would be `"chicken_cry"`. Whether these asset files exist is not validated; missing audio silently fails (Bevy's asset system will warn but not crash).

### GAP-02 — Feed trough proximity not checked
`handle_feed_trough_interact` feeds all productive animals whenever **any** hay is removed from inventory, regardless of the player's proximity to the trough. The trough entity exists and has a `FeedTrough` component, but the system never queries the trough's position or checks player distance. Hay consumed anywhere on the map feeds all animals.

### GAP-03 — No animal cap per building level
`animal_state.coop_level` and `barn_level` exist in `AnimalState` but `handle_animal_purchase` only checks `has_coop`/`has_barn` (bool). There is no enforcement of a maximum animal count per building tier.

### GAP-04 — Happiness never decreases from petting skip
Not petting an animal does not reduce happiness — only not feeding does. An animal that is fed daily but never petted will stay at max happiness (happiness only decreases from lack of feeding). This means petting has asymmetric impact.

### GAP-05 — `AnimalState.animals` rebuilt every frame
`sync_animal_state_resource` clones and rebuilds the full `Vec<Animal>` every frame. With < 20 animals this is cheap, but it's a potential scaling concern and discards any in-flight mutations from systems that run after the sync in the same frame.
